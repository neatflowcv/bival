# ADR 0004: `Unknown Object`는 정말 알 수 없는 경우에만 사용한다

- 상태: 승인됨
- 날짜: 2026-04-09

## 배경

현재 `analyze`는 엔트리 그룹이 `Unversioned Object`나 `Versioned Object` 규칙을 만족하지 않으면 넓은 의미로 `Unknown Object`로 취급한다.

하지만 이 범주에는 성격이 다른 두 경우가 함께 섞여 있다.

- 정말로 어떤 오브젝트 형태인지 추론할 수 없는 경우
- 버저닝 오브젝트 또는 비버저닝 오브젝트로 보이지만, 일부 엔트리가 빠졌거나 상호 일관성이 깨져서 규칙을 만족하지 못한 경우

이 둘을 같은 `Unknown Object`로 묶으면 실제 데이터 문제를 분석하기 어렵다.

- 누락된 엔트리인지
- 개수 불일치인지
- head/plain/instance/olh의 대응 관계가 깨졌는지
- pending 상태 때문에 아직 안정 상태가 아닌지

위와 같은 차이를 출력에서 구분할 수 없기 때문이다.

## 결정

프로젝트는 앞으로 `Unknown Object`를 마지막 보류 분류로만 사용한다.

즉, `analyze`가 어떤 그룹을 처리할 때는 아래 순서를 따른다.

1. 먼저 `Unversioned Object`와 `Versioned Object`인지 판별한다.
2. 판별에 실패하면, 알려진 패턴의 불일치인지 분석한다.
3. 알려진 불일치라면 `Unknown Object`로 뭉개지지 않고 원인 중심으로 진단한다.
4. 어떤 기존 패턴으로도 설명할 수 없을 때만 `Unknown Object`를 사용한다.

여기서 `Unknown Object`는 "규칙 불일치 전체"를 뜻하지 않는다.
`Unknown Object`는 "현재 모델과 규칙으로도 의미를 설명할 수 없는 잔여 집합"을 뜻한다.

## 근거

- `analyze`의 목적은 단순 분류보다 실제 BI 엔트리 상태를 해석하는 데 있다.
- 이미 알려진 구조 불일치를 `Unknown Object`로 숨기면 후속 분석 비용이 커진다.
- 데이터 품질 문제를 해결하려면 실패 원인이 구체적으로 드러나야 한다.
- `Unknown`은 분류기의 편의상 넓게 쓰기 쉽지만, 실제 운영과 디버깅에서는 정보 손실이 크다.

## 결과

긍정적 영향은 아래와 같다.

- `analyze` 출력이 조사 가능한 문제를 더 직접적으로 보여준다.
- 새로운 BI 패턴과 단순 데이터 이상을 구분하기 쉬워진다.
- `Unknown Object`가 실제 미지 패턴 탐지 신호로서 더 신뢰 가능해진다.

주의사항은 아래와 같다.

- 분류 실패 경로에 대한 진단 규칙을 별도로 관리해야 한다.
- 새로운 진단 이유가 추가되면 테스트와 문서도 함께 갱신해야 한다.
- 어떤 경우를 "알려진 불일치"로 볼지에 대한 기준이 흐려지지 않도록 유지해야 한다.

## 구현 지침

- `analyze`는 문제 그룹을 출력할 때 가능하면 구조적 원인을 직접 기록한다.
- 개수 불일치, head 누락, pair 불일치, OLH 참조 불일치처럼 이미 해석 가능한 실패는 진단 가능한 문제로 취급한다.
- pending 엔트리는 이미 별도 상태로 보고하므로, `Unknown Object` 판정 전에 우선 처리한다.
- 새로 발견된 `Unknown Object` 사례는 샘플을 확보하고, 가능한 경우 새로운 규칙 또는 진단 사유로 흡수한다.

## 고려했으나 채택하지 않은 대안

### 기존처럼 모든 규칙 불일치를 `Unknown Object`로 유지

장점:

- 구현이 단순하다.
- 분류기 인터페이스를 최소한으로 유지할 수 있다.

단점:

- 알려진 실패와 미지 패턴을 구분할 수 없다.
- `analyze` 결과만으로는 후속 조치 방향을 정하기 어렵다.
- 같은 유형의 문제를 반복해서 수동 분석하게 된다.

위 이유로 채택하지 않았다.
