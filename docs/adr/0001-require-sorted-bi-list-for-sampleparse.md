# ADR 0001: sampleparse 입력으로 정렬된 BI List를 강제한다

- 상태: 승인됨
- 날짜: 2026-03-24

## 배경

기존에는 `sampleparse`가 `radosgw-admin bi list` 결과를 그대로 입력으로 사용한다고 가정했다.

하지만 실제 처리 과정에서는 같은 `name`을 가진 레코드들끼리 묶어서 해석해야 하는 연산이 자주 필요하다. 원본 BI list는 같은 `name`의 엔트리들이 떨어져 있을 수 있으므로, 단순 스트리밍 처리만으로는 동일 오브젝트의 관련 엔트리를 안정적으로 함께 다루기 어렵다.

이 문제를 원본 입력 그대로 유지한 채 해결하려면 아래와 같은 비용이 생긴다.

- 전역 버퍼링 또는 대규모 인메모리 인덱싱이 필요하다.
- 입력 순서에 따라 처리 로직이 복잡해진다.
- 같은 `name` 기준의 검증과 집계 로직이 일관되지 않다.

## 결정

`sampleparse`의 입력은 더 이상 원본 `bi list`가 아니라, `bisort`를 통해 `name` 기준으로 정렬된 BI list로 강제한다.

즉, 처리 파이프라인은 아래 순서를 전제로 한다.

1. `radosgw-admin bi list`로 원본 BI list를 추출한다.
2. `bisort`로 같은 `name`의 레코드들이 인접하도록 정렬한다.
3. `sampleparse`는 이 정렬된 결과만 입력으로 받는다.

`sampleparse`는 입력이 이미 정렬되어 있다는 전제 위에서 동일 `name` 그룹 단위의 연산을 수행한다.

## 근거

- 같은 `name` 레코드가 인접하면 그룹 단위 스트리밍 처리가 가능하다.
- 메모리 사용량을 전체 입력 크기보다 현재 그룹 크기에 가깝게 제한할 수 있다.
- 검증, 집계, 상호 비교 로직을 단순화할 수 있다.
- 입력 계약을 명확히 분리할 수 있다.
  - `bisort`: 정렬 책임
  - `sampleparse`: 그룹 해석 및 검증 책임

## 결과

긍정적 영향은 아래와 같다.

- `sampleparse` 구현이 단순해진다.
- 동일 오브젝트 단위의 검증 로직을 안정적으로 작성할 수 있다.
- 대용량 입력에서도 스트리밍 친화적인 구조를 유지할 수 있다.

부정적 영향 및 주의사항은 아래와 같다.

- 사용자는 `sampleparse` 전에 반드시 `bisort`를 수행해야 한다.
- 정렬 단계가 추가되므로 처리 시간이 늘 수 있다.
- 정렬되지 않은 입력을 `sampleparse`에 직접 넣으면 결과가 불완전하거나 잘못될 수 있다.

## 고려했으나 채택하지 않은 대안

### 원본 BI List를 그대로 사용

장점:

- 전처리 단계가 없다.
- 사용법이 단순해 보인다.

단점:

- 같은 `name`끼리 묶는 연산이 어렵다.
- 전체 입력을 기억하거나 복잡한 상태 관리를 해야 한다.
- 구현 복잡도와 오류 가능성이 커진다.

위 이유로 채택하지 않았다.
